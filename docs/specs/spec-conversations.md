# 规格：会话与消息

## 基本信息

- 名称：会话与消息
- 负责人：未指定
- 创建日期：2026-01-30
- 最近更新：2026-02-11

## 背景

- 需要以会话为单位组织对话与代码修改。

## 目标

- 支持创建会话、发送消息、流式响应。
- 通过状态控制写操作。

## 非目标

- 不支持会话恢复为 ACTIVE（归档不可逆）。

## 范围

- In：会话创建、消息发送、归档、可见性。
- Out：复杂多状态机。

## 业务规则

- 会话状态只有 ACTIVE / ARCHIVED。
- 归档后禁止发送消息与创建 MR/预览。
- 可见性为 PRIVATE / PUBLIC。

## 需求

### 功能需求

- F1：创建会话并保存上下文。
- F2：消息通过 SSE 返回。
- F3：会话可归档。
- F4：可更新会话可见性。
- F5：支持中断流式响应。
- F6：当消息包含图片附件时，后端必须先调用视觉模型提炼图片关键信息，再将“原始文本 + 图片关键信息”汇总后交由主 AI 执行。

### 非功能需求

- N1：消息响应需支持流式输出。
- N2：对话列表接口服务端响应时间需满足性能测试阈值。
- N3：图片两阶段处理需在服务端日志中输出视觉模型提炼的特征内容，便于排障。
- N4：会话列表与消息历史查询使用“分步查询 + 内存合并”，不使用跨表连接查询。
- N5：会话详情/列表/消息历史/上下文读取采用进程内 LRU 缓存，写操作后需清理对应缓存键。
- N6：缓存异常不得影响会话主流程可用性，读取链路需自动降级为数据库直读。
- N7：`message_metadata.images` 不进入缓存，图片附件仅存储于数据库。
- N8：会话相关接口响应日志需记录方法、路径、状态码与耗时，便于线上性能排查。
- N9：`GET /api/conversations/:sessionId/messages` 必须支持可选参数 `since`（ISO 时间），用于增量拉取消息，避免频繁全量查询。
- N10：`GET /api/conversations` 的用户可见性与环境过滤必须在存储查询阶段下推（数据库筛选），禁止先全量加载再在内存过滤。
- N11：`GET /api/conversations/:sessionId/messages` 在 `since` 为空且请求携带 `If-None-Match` 时，必须先执行轻量版本校验；未变更需直接返回 `304`，禁止先加载全量消息再判定未变更。
- N12：缓存访问失败后，读取链路需直接走无缓存路径，避免单次请求被缓存层阻塞拖慢。
- N13：`GET /api/conversations/:sessionId` 的会话基础信息与上下文读取必须并行执行，禁止串行等待导致时延叠加。
- N14：同一进程内并发读取同一 `sessionId` 详情时，必须合并为单次存储层读取，禁止重复查询数据库。
- N15：`GET /api/conversations` 必须返回会话页首屏所需的完整会话字段（脱敏后），前端进入会话页时应直接复用该数据，禁止首屏重复请求 `GET /api/conversations/:sessionId`。
- N16：`GET /api/conversations/:sessionId/messages` 的消息与元数据读取必须使用单次 SQL（`messages` + `message_metadata` 左连接）并依赖 `(conversation_id, timestamp)` 复合索引，禁止先查消息再二次 `IN` 查询元数据。
- N17：`GET /api/conversations/:sessionId/messages` 必须输出分阶段耗时日志（会话校验、版本校验、消息读取、总耗时）与结果规模（消息条数），便于线上慢请求定位。
- N18：`GET /api/conversations/:sessionId/messages` 的权限校验必须走轻量会话访问查询（仅 `id/user_id/visibility`），禁止为鉴权加载完整会话上下文。
- N19：数据库访问层必须提供统一 SQL 日志出口（至少包含 SQL 文本摘要与参数规模），用于跨接口性能排查与问题回放。

## 用户体验

- 发送消息立即看到用户输入与“思考中”。
- 流式返回增量展示。
- 创建对话选择项目后不展示仓库与工作目录。
- 连续对话底部输入区与模型切换、发送按钮保持同一高度与视觉风格。
- 新建对话输入区的图片上传按钮位置需与连续对话输入区一致，统一放在输入操作区右侧；选图后的附件预览需渲染在输入框区域内，不在输入框下方单独占位。
- 新建对话页卡片、筛选控件、输入框与底部操作区视觉需保持统一（圆角、边框、阴影和间距一致）；创建按钮文案为“创建对话”，对话模式分段选项需平分并铺满可用宽度，保证对齐与可读性；创建按钮默认常亮展示，空内容点击后通过校验提示阻断提交；底部上传/模型/创建操作区需按内容宽度收敛并右对齐，不保留大块空白占位。
- 新建对话底部上传按钮、模型选择器、创建按钮需保持同一垂直中线对齐，避免控件内部文本与箭头出现视觉错位。
- 新建对话底部上传按钮与模型选择器尺寸需适度收敛，避免操作区控件视觉过重。
- 新建对话附件缩略图删除按钮需使用小尺寸内嵌样式，不能越出缩略图边界或被裁剪。
- 新建对话与连续对话输入框需支持粘贴图片，粘贴后按现有附件流程渲染预览并随消息提交。
- 移动端创建对话使用独立页面，按项目/分支/模式/模型/需求输入引导完成创建。
- 创建对话阶段支持上传图片附件，并在会话创建成功后随首次自动发送消息一并提交。
- 对话列表信息层级清晰，标题单行截断，操作区与状态标识对齐，避免视觉拥挤。

## 数据与接口

- conversations / conversation_contexts / messages / message_metadata。
- `message_metadata` 必须包含 `images`（jsonb）字段，用于持久化图片附件元数据。
- API：/api/conversations 及子路由（含 /:sessionId/interrupt）。
- `GET /api/conversations/:sessionId/messages` 支持 `since` 查询参数；无 `since` 返回全量历史，有 `since` 仅返回该时间点之后的消息。
- 会话链路接口（创建/列表/详情）返回的 `projectInfo` 不包含 `gitRepositoryUrl` 字段。

## 验收标准

- A1：ACTIVE 会话可发送消息。
- A2：ARCHIVED 会话发送消息返回 403。
- A3：可见性切换成功。
- A4：中断请求会终止流式响应。
