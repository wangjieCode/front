# 迭代记录：2026-03-04

## 背景

- 现有 `LruCacheService` 仅支持进程内缓存，不支持重启后恢复。
- 需要为缓存增加定时持久化能力，降低服务重启导致的缓存冷启动影响。

## 变更

- `backend/src/services/LruCacheService.ts`
  - 新增缓存快照结构与持久化流程。
  - 支持环境变量驱动的定时持久化：
    - `LRU_CACHE_PERSIST_PATH`
    - `LRU_CACHE_PERSIST_INTERVAL_MS`（默认 `60000`）
  - 支持服务启动自动恢复缓存快照。
  - 新增 `persistNow()` 用于手动立即落盘。
- `backend/src/index.ts`
  - 在 `SIGTERM` 与 `SIGINT` 优雅退出流程中，新增 `LruCacheService.persistNow()`。
- 线上环境配置补充：
  - `backend/.env.production` 增加 `LRU_CACHE_PERSIST_PATH` 与 `LRU_CACHE_PERSIST_INTERVAL_MS`。
  - `LRU_CACHE_PERSIST_INTERVAL_MS` 调整为 `3600000`（1 小时）。
- 文档同步：
  - `docs/specs/spec-conversations.md`
  - `docs/context/environment.md`
  - `docs/technical/backend_services.md`

## 影响

- 缓存从“纯内存临时态”升级为“可恢复态”，服务重启后可回填缓存，减少短时查询抖动。
- 不影响 Redis 队列/定时任务链路。

## 验证

- 执行 `pnpm -C backend build` 通过。
- 代码检视确认缓存写盘采用临时文件 + 原子重命名，避免写盘中断导致快照损坏。
- 检查 `backend/.env.production` 已包含 LRU 持久化配置项。

## 风险

- 持久化文件位于本地磁盘，若运行环境为无状态容器且未挂载持久卷，重启后仍可能丢失缓存快照。

### 风险

- 永不过期后，缓存一致性更依赖“写后删键”；若存在漏删路径，旧数据会长期驻留。
- 在 `50MB` 上限下，旧热点会更依赖 LRU 淘汰，不再依赖 TTL 自然过期，可能影响新热点命中结构。
