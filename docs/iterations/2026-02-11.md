## 本次追加（会话/消息接口去连接查询优化）

### 背景

- `GET /api/conversations` 与 `GET /api/conversations/:sessionId/messages` 依赖连接查询，数据量上来后响应明显变慢。

### 变更

- `backend/src/storage/DrizzleConversationStorage.ts`
  - `listSessions` 移除 `conversations + projects + conversation_contexts` 连接查询。
  - 改为两次查询：先查 `conversations`，再按 `conversation_id` 批量查 `conversation_contexts`，在内存中合并结果。
  - `loadSession` 移除 `conversations + projects` 连接查询，改为单表查询 `conversations` + 单独读取上下文。
  - `loadMessagesWithMetadata` 移除 `messages + message_metadata` 连接查询。
  - 改为两次查询：先按会话查消息，再按 `message_id` 批量查元数据，在内存中合并。
- `backend/src/storage/ConversationStorageAdapter.ts`
  - 删除对连接查询回填字段 `projectNameJoined/projectRepoUrl` 的依赖，统一使用会话与上下文自身字段。
- `docs/specs/spec-conversations.md`
  - 新增非功能约束：会话列表与消息历史查询采用“分步查询 + 内存合并”，不使用跨表连接查询。

### 影响

- 消息历史与会话列表查询路径更稳定，避免连接查询在高基数下的开销放大。
- 查询逻辑更直接，可维护性更高，后续定位性能瓶颈更容易。
- 不再依赖 `projects` 连接回填会话字段，数据来源更单一。

### 验证

- 通过代码检视确认三个核心读取路径已去除 join：`loadSession`、`listSessions`、`loadMessagesWithMetadata`。
- 自动化测试在本次改动后执行（见本次提交说明）。

### 风险

- 去除 `projects` 回填后，历史会话若未持久化完整项目信息，`gitRepositoryUrl` 将返回空字符串（当前前端主流程不依赖该字段）。

## 本次追加（Codex 探测 ESM 加载兼容修复）

### 背景

- 线上日志出现 `require() cannot be used on an ESM graph with top-level await`。
- 后端以 CommonJS 编译运行时，`loadNeovateSdk()` 对 `@neovate/code` 的加载在产物中被降级为 `require()`，导致 Codex 启动探测失败并被标记为不可用。

### 变更

- `backend/src/utils/NeovateSdkRunner.ts`
  - `loadNeovateSdk()` 改为运行时动态 `import()`（通过 `Function` 间接调用），避免 TypeScript 在 CommonJS 目标下将其编译为 `require()`。
  - 保留现有会话创建、恢复与执行链路，不引入兼容分支。
- `docs/specs/spec-model-selection.md`
  - 补充后端行为约束：Neovate SDK 必须以运行时动态 `import()` 加载，避免 ESM 包在 CJS 产物中加载失败。

### 影响

- 线上 `ModelAvailabilityService` 可正常加载 `@neovate/code` 并执行 Codex 可用性探测。
- Codex 未认证或网络异常时仍会被正确判定不可用；不再因模块系统不兼容误判。

### 验证

- 运行 `pnpm --dir backend build`，确认构建通过。
- 检查 `backend/dist/utils/NeovateSdkRunner.js`，确认不再出现 `require('@neovate/code')` 的加载路径。

### 风险

- 该修复依赖运行时支持标准 `import()`；当前 Node 运行环境满足该前提。

### 变更

- `deploy.mjs`
  - API 启动命令改为 PM2 直接启动 `src/index.ts`：
    - `pm2 start src/index.ts --name <app> -i <instances> --interpreter node --interpreter-args="--import tsx" --update-env`
  - 运行时校验 `check_runtime` 改为校验：
    - `exec_mode=cluster_mode`
    - `exec_interpreter=node`
    - `pm_exec_path` 以 `/src/index.ts` 结尾
    - `node_args` 或 `interpreter_args` 包含 `--import tsx`
  - 部署前入口检查从 `node_modules/tsx/dist/cli.mjs` 改为 `src/index.ts` 存在性检查。
  - 部署失败日志增加 `~/.pm2/pm2.log` 最近 120 行输出，避免应用日志为空时无排障信息。
- `docs/specs/spec-monorepo-workspace.md`
  - 增补 API 多实例部署约束，明确使用 cluster 直启，不走 `tsx` CLI 中转。

### 影响

- API 多实例可共享同一监听端口，避免 `EADDRINUSE` 触发的重启死循环。
- 旧运行时（`tsx` CLI 中转）会被识别为不符合预期并自动重建为新模式。

### 验证

- `node --check deploy.mjs` 语法检查通过。
- 代码检视确认 API 启动与运行时校验均已切换到 cluster 直启模型。

### 风险

## 本次追加（消息历史接口慢查询优化）

### 背景

- 线上 `GET /api/conversations/:sessionId/messages` 出现 7s+ 慢请求。
- 存储层原实现为“两次查询”：
  - 先查 `messages`；
  - 再按消息 ID 列表查询 `message_metadata` 并在内存合并。
- 在消息量大时，第二次 `IN (...)` 查询和应用层合并会放大耗时。

### 变更

- `backend/src/storage/DrizzleConversationStorage.ts`
  - `loadMessagesWithMetadata` 改为单次 SQL：
    - `messages` 左连接 `message_metadata`；
    - 按 `messages.timestamp` 排序；
    - 移除“先查消息再 `IN` 查元数据”的双查询路径。
- `backend/src/db/schema.ts`
  - `messages` 新增复合索引：`idx_messages_conversation_timestamp (conversation_id, timestamp)`。
- `backend/drizzle/0005_messages_conversation_timestamp_index.sql`
  - 新增数据库迁移，创建上述复合索引。
- `backend/drizzle/setup-supabase.sql`
  - 初始化脚本同步添加 `idx_messages_conversation_timestamp`。
- 文档同步：`docs/specs/spec-conversations.md`
  - 新增消息历史查询必须“单次左连接 + 复合索引”的非功能约束。

### 影响

- 消息历史查询减少一次数据库往返与大 `IN` 列表解析开销。
- `(conversation_id, timestamp)` 复合索引可优化会话内按时间排序读取路径。

### 验证

- `pnpm --filter web-frontend-intern-assistant-backend build` 通过。

### 风险
