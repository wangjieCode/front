# 迭代记录：2026-02-09

## 背景

- 平台需要提供 Codex 可用的 GPT 系列模型，同时保留既有 IFLOW 模型选择能力。

## 变更

- 前端模型选项改为双集合并：保留 IFLOW 模型，并新增 Codex GPT 系列（`codex/gpt-5-codex`、`codex/gpt-5`、`codex/gpt-5-mini`、`codex/gpt-4.1`、`codex/gpt-4.1-mini`）。
- 后端模型白名单同步改为双集合并，支持 `iflow/*` 与 `codex/*`。
- 默认模型统一调整为 `codex/gpt-5-codex`。
- 更新模型选择规格文档与 SDK 验证脚本规格文档，默认模型与支持列表同步为双模型体系。
- 新增后端模型配置接口 `GET /api/conversations/models`，返回默认模型与每个模型的可用状态（`enabled`）。
- 启动阶段新增 Codex Auth 探测：调用 `codex/gpt-5-codex` 最小请求验证可用性；失败时自动禁用全部 `codex/*` 模型选项。
- 前端模型下拉切换为读取后端配置，`enabled=false` 的模型在下拉中置灰禁选。
- 新增后端单元测试 `modelAvailabilityService.test.ts`，覆盖 Codex Auth 探测成功/失败下的模型启用状态与默认模型回退逻辑。
- 修复“Codex 会话卡在 AI 思考中”问题：消息发送阶段统一解析并强制传递可用模型（优先请求模型，否则会话模型，不可用自动回退默认可用模型）。
- 为 Neovate SDK 执行增加统一超时控制（`NEOVATE_EXEC_TIMEOUT_MS`，默认 180000ms），超时后中断并返回错误，避免 SSE 长时间悬挂。
- 修复前端“AI 正在处理您的消息...”卡住：SSE `reader` 结束时增加强制收尾（兜底 `markStreamComplete`），并将该提示仅限 `isStreaming=true` 时显示。
- 新增前端对 `chunk` 中 `{\"type\":\"result\",\"isError\":true}` 的错误识别：直接将错误内容渲染为消息文本（如 `Cannot read properties...`），并立即结束流式状态。
- 优化 `result/isError` 错误文案：将 `Cannot read properties of null (reading 'model')` 映射为可读提示（模型会话异常，建议重试或切换模型）。
- 修复错误消息“刷新后丢失”：`MessageList` 在回放历史 `stream-json` 时可解析 `result/isError` 并稳定展示错误文本。
- 修复 Codex 会话恢复异常：当 `resumeSession` 失败时自动降级为 `createSession` 重试；并允许对话会话 ID 映射更新，覆盖旧的失效会话 ID。
- 修复 SDK 成功判定：`runNeovateSdk` 识别 `result/isError`（或 `subtype=error`）并返回错误，避免将“业务错误结果”误判为可用。

## 影响

- 新建对话与会话消息发送时，可选择并透传 IFLOW 或 Codex GPT 模型。
- 既有 IFLOW 模型参数继续可用，不会因本次变更失效。
- 前后端模型枚举与文档保持一致，减少配置歧义。
- Codex OAuth 未登录或失效时，用户界面不再允许选择 `codex/*` 模型，避免无效请求。

## 验证

- 检查 `spec-model-selection` 与 `spec-neovate-sdk-verify` 默认模型已同步为 `codex/gpt-5-codex`。
- 检查后端 `GET /api/conversations/models` 已返回 `enabled` 状态，并在前端下拉生效。

## 风险

## 补充迭代（登录态字段收敛）

### 背景

- 登录态维护已调整，继续沿用旧 localStorage 字段可能导致误判为已登录。

### 变更

- 前端认证存储字段改为新命名：`fi_auth_user_id_v2`、`fi_auth_username_v2`、`fi_auth_has_password_v2`、`fi_auth_token_v2`。
- 登录态判定改为必须包含 `fi_auth_token_v2`。
- 所有认证请求头注入统一改为 `authUtils.getAuthHeaders()`，移除业务代码中直接读取 localStorage 的旧字段逻辑。
- 同步更新认证相关规格与技术文档。

### 影响

- 旧字段残留不会再触发前端登录态误判。
- 认证逻辑收敛到 `authUtils`，后续维护成本降低。

### 验证

- 全局检索确认前端业务代码无 `user_id`/`username` 旧字段读写残留。
- 单测初始化登录态改为调用 `authUtils.setUserInfo`，与线上路径一致。

### 风险

- 变更后用户需重新登录一次以写入新字段。

## 补充迭代（老账号密码初始化）

### 背景

- 已存在但无密码哈希的历史账号，调用 `POST /api/auth/login` 时不会写入密码，导致响应 `hasPassword=false`。

### 变更

- 调整后端登录逻辑：若账号存在且 `passwordHash` 为空，强制校验本次提交密码并写入 `password_hash`。
- 登录成功响应中的 `hasPassword` 改为基于本次流程结果计算，确保初始化后立即返回 `true`。
- 同步更新认证规格文档，补充该场景的业务规则与验收标准。

### 影响

- 历史无密码账号首次登录会完成密码初始化，后续登录走密码校验。
- 消除“登录成功但未设置密码”的状态不一致问题。

### 验证

- 代码路径确认：`existing user + empty passwordHash` 分支现在会写入 `password_hash`。
- 接口行为预期：首次初始化后响应 `hasPassword=true`。

### 风险

- 历史账号首次初始化密码后，必须使用该密码登录，忘记密码需通过改密流程处理。

## 补充迭代（鉴权切换为 JWT）

### 背景

- 现有鉴权依赖 `x-user-id` / `x-username` 自定义头，不符合标准 JWT 鉴权实践。

### 变更

- 新增后端 JWT 工具，登录成功后签发 token，并在响应中返回 `token`。
- `requireAuth` 中间件改为校验 `Authorization: Bearer <jwt>`，并通过 `sub` 关联用户。
- `/api/auth/verify`、`/api/auth/users/:id/password`、`/api/auth/users/:id/username` 改为统一复用 `requireAuth`。
- 前端 `authUtils.getAuthHeaders()` 改为注入 Bearer token，移除 `x-user-id`/`x-username` 透传。
- 移动端登录补齐密码输入并统一走 JWT 登录返回。
- 同步更新认证规格与技术文档。

### 影响

- 所有受保护 API 现在只接受 Bearer JWT。
- 旧的自定义用户头鉴权路径被移除。

### 验证

- 代码检索确认前端业务请求不再设置 `x-user-id`/`x-username`。
- 登录成功链路包含 `token` 返回并写入 `fi_auth_token_v2`。

### 风险

- 未配置 `JWT_SECRET` 时将使用开发默认值；生产环境需显式配置强随机密钥。

- 调整项目列表缓存清理策略：写操作后清理 `projects:list:*` 全局列表缓存。

### 验证

- 代码路径确认 `getProjects` 不再使用 `ownerId` 作为筛选条件。

### 风险

## 补充迭代（JWT 改为社区标准库实现）

### 背景

- 认证链路虽使用 JWT 格式，但签发/验签逻辑为手写实现，不符合“使用社区标准方案”的要求。

### 变更

- 后端新增依赖：`jsonwebtoken`、`@types/jsonwebtoken`。
- 重写 `backend/src/utils/jwt.ts`：改为使用 `jsonwebtoken` 进行 JWT 签发与校验。
- 保持现有接口契约不变：token 结构、默认有效期（7 天）、Bearer 提取方式保持一致。
- 同步更新认证规格与技术文档。

### 影响

- JWT 逻辑从自研实现切换为社区标准库实现，维护性与兼容性更好。

### 验证

- 代码路径确认 `signAuthToken`/`verifyAuthToken` 已切换到 `jsonwebtoken`。
- 认证中间件与登录接口无需改动调用方式。

### 风险

- 依赖新增后，部署环境需保证安装后端新依赖。

## 补充迭代（登录后刷新项目列表）

### 背景

- 用户在未登录状态进入项目页后，登录成功不会自动重新请求项目列表接口，页面可能停留在旧状态。

### 变更

- 登录成功时在前端广播统一事件 `fi:login-success`。
- `ProjectsPage` 与 `MobileProjectsPage` 监听该事件并自动调用项目列表加载。
- `ProjectSelector` 与 `MobileProjectSelector` 同步监听该事件，登录后自动刷新可选项目数据。
- 同步更新项目管理规格文档。

### 影响

- 登录成功后，无需手动刷新页面即可看到最新项目列表。

### 验证

- 在未登录时进入项目页，完成登录后项目列表会自动请求并更新。
- 项目选择器在登录成功后会自动刷新数据源。

### 风险

- 事件驱动刷新依赖浏览器全局事件，后续若拆分微前端需统一事件命名约定。
