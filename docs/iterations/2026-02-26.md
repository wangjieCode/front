# 迭代记录：2026-02-26

## 背景

- 当前业务缓存链路仍保留 Redis 命名与依赖残留，不符合“除定时任务外不使用 Redis 缓存”的目标。
- 需要统一业务缓存实现并明确内存上限，避免缓存容量无限增长。

## 变更

- 业务缓存服务重构为 `LruCacheService`，统一使用进程内 LRU：
  - 新增：`backend/src/services/LruCacheService.ts`
  - 删除：`backend/src/services/RedisCacheService.ts`
- 统一替换业务缓存调用方：
  - `backend/src/services/ConversationManager.ts`
  - `backend/src/services/ProjectService.ts`
  - `backend/src/services/WorktreeManager.ts`
  - `backend/src/storage/DrizzleConversationStorage.ts`
- 移除会话管理器中的 Redis 参数残留与相关日志文案，精简构造签名。
- 删除未使用的 Redis 连接管理实现：
  - 删除：`backend/src/db/RedisManager.ts`
- 删除后端包中仅用于业务缓存的 Redis 依赖：
  - `backend/package.json` 移除 `ioredis` 与 `@types/ioredis`
- 缓存容量约束：
  - 统一将进程内缓存总上限设置为 `50MB`（`maxSize` + `sizeCalculation`）。
- 文档同步：
  - 更新规格：`docs/specs/spec-conversations.md`、`docs/specs/spec-projects.md`、`docs/specs/spec-worktree.md`
  - 更新技术文档：`docs/technical/backend_services.md`、`docs/technical/data_model.md`
  - 更新环境变量文档：`docs/context/environment.md`（移除 `DISABLE_REDIS`）

## 影响

- 业务读缓存完全去 Redis 化，缓存路径与故障面收敛到单一进程内实现。
- 定时任务/BullMQ 仍通过 Redis 运行，不受本次改动影响。
- 缓存内存占用可控，避免超预期膨胀。

## 验证

- 全局检索确认业务代码不再存在 `RedisCacheService`、`RedisManager`、`ioredis` 直接引用。
- 执行 `pnpm -C backend build` 通过。

## 风险

- 业务缓存为进程内实现，多实例部署下缓存不共享，命中率可能低于集中式缓存。
- `pnpm install --lockfile-only` 因网络不可达未完成，锁文件未在本次迭代中更新。
