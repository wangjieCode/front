# 单元测试技术选型

> 范围：当前仓库的前端（React + Vite）与后端（Node.js + Express + TypeScript）。

## 选型结论（先落地、再迭代）

### 后端（backend）
- **测试框架：Jest**（已有依赖与脚本）。
- **断言/Mock**：Jest 内置即可满足；如需 HTTP 层可继续用 `supertest`（已在依赖中）。
- **TypeScript 支持**：`ts-jest` 已配置依赖，可直接运行。

**理由**
- 当前 `backend/package.json` 已内置 `jest` / `ts-jest` / `@types/jest` 与 `supertest`，零成本启用，减少迁移风险。
- 适配现有脚本（`pnpm test`、`pnpm test:watch`），无需改动 CI 或开发习惯。

### 前端（frontend）
- **测试框架：Jest**（已有 `test` 脚本）。
- **组件测试：React Testing Library（建议新增）**。
- **DOM 环境：jsdom（随 Jest 使用）**。

**理由**
- 当前前端 `package.json` 已提供 `jest` 入口，延续一致性和较低的工具切换成本。
- React Testing Library 是 React 生态主流，利于关注“用户行为”而非实现细节。

> 说明：若后续需要更快的本地执行速度或更强的 ESM 兼容性，可评估 Vitest；但短期以“最少变更、最快落地”为原则。

## 关键节点覆盖优先级（单元测试层）

> 单元测试优先覆盖“输入校验/状态流转/权限处理”这类高风险逻辑。

### 后端（示例）
1. **会话创建参数校验**：`initialPrompt`/`projectId`/`mode` 校验逻辑。
2. **编辑/只读模式关键链路**：
   - 创建会话时根据 mode 建立 **正确的 worktree**。
   - worktree 路径与会话上下文存储一致（写入正确的 `worktreePath`/`workDir`）。
   - 只读模式禁止编辑动作；编辑模式允许写入。
   - AI 服务可用于对话/编辑（Mock 或本地桩，验证调用参数/次数）。
3. **预览部署状态切换**：BUILDING/RUNNING/ERROR 逻辑。
4. **Docker 操作参数校验**：必要参数缺失时的返回。

### 前端（示例）
1. **请求鉴权头拼装**（`user_id`/`username`）。
2. **401 处理**（清理本地存储 & 触发登录弹窗）。
3. **轮询策略参数**（活跃/降频/超时）。
4. **流式响应展示**：
   - **创建对话后首次发送消息**的流式返回是否正确逐步渲染。
   - **连续对话**的流式消息是否在列表中追加并保持顺序。
   - 请求失败时的 UI 回退/错误提示是否出现。

## 后续扩展方向（可选）
- **快照/视觉回归**：不建议在单元层引入，可放到 E2E 或 Storybook 回归。
- **属性测试（property-based testing）**：后端已有 `fast-check` 依赖，可在关键业务逻辑加入属性测试。
- **统一覆盖率门槛**：待测试案例稳定后再引入，避免阻塞早期改造。

## 落地步骤（最低成本）
1. 新增基础测试目录与示例用例（后端/前端各 1-2 个）。
2. 验证 `pnpm test` 能在本地与 CI 运行。
3. 按关键节点逐步补齐用例。
