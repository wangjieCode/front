# 代码工具可配置化 - 需求文档

## 简介

本系统是对现有 Web 前端实习生助手系统的架构优化，将硬编码的 qodercli 工具改为可配置的代码工具系统。通过引入适配器模式，系统能够支持多种 AI 代码工具（qodercli、Cursor、Copilot 等），提高系统的灵活性和可扩展性。同时增强项目管理能力，支持从零创建新工程，并在功能完成后自动清理分支。

## 术语表

- **代码工具**: 用于执行代码修改的 AI 工具，如 qodercli、Cursor、Copilot 等
- **工具适配器**: 统一不同代码工具接口的抽象层，使系统能够支持多种工具
- **工具提供者**: 实现特定代码工具接口的具体类，如 QoderCliProvider、CursorProvider
- **项目初始化**: 在工作目录中克隆 Git 仓库并设置工作环境的过程
- **工作目录**: 虚拟机上存放所有项目代码的根目录
- **功能分支**: 为特定功能创建的 Git 分支，完成后可合并或删除

## 需求

### 需求 1: 可配置代码工具

**用户故事:** 作为系统管理员，我想要配置不同的 AI 代码工具，以便根据项目需求选择最合适的工具

#### 验收标准

1. WHEN 系统启动 THEN 系统应从配置文件读取代码工具类型和参数
2. WHEN 配置文件指定 qodercli THEN 系统应使用 qodercli 执行代码修改
3. WHEN 配置文件指定其他工具 THEN 系统应使用对应的工具适配器执行代码修改
4. WHEN 代码工具未配置 THEN 系统应使用默认工具或提示用户配置

### 需求 2: 项目初始化功能

**用户故事:** 作为用户，我想要在工作目录中初始化已有的 Git 项目，以便在指定的代码库上工作

#### 验收标准

1. WHEN 用户请求初始化项目 THEN 系统应询问 Git 仓库地址和分支信息
2. WHEN 用户提供 Git 仓库地址 THEN 系统应在虚拟机工作目录中克隆该仓库
3. WHEN 克隆仓库 THEN 系统应验证仓库是否成功克隆并检查项目结构
4. WHEN 用户指定分支 THEN 系统应切换到指定分支
5. WHEN 项目初始化完成 THEN 系统应将项目路径保存到会话上下文，后续操作使用该项目

### 需求 3: 项目管理

**用户故事:** 作为用户，我想要管理工作目录中的多个项目，以便在不同项目间切换工作

#### 验收标准

1. WHEN 用户查询项目列表 THEN 系统应显示工作目录中所有已初始化的项目
2. WHEN 用户切换项目 THEN 系统应更新当前工作项目上下文
3. WHEN 用户查询当前项目 THEN 系统应显示当前工作的项目名称和路径
4. WHEN 用户删除项目 THEN 系统应在确认后删除工作目录中的项目文件夹
5. WHEN 项目不存在 THEN 系统应提示用户初始化新项目或选择其他项目

### 需求 4: 分支生命周期管理

**用户故事:** 作为用户，我想要在功能完成后自动清理分支，以便保持代码仓库整洁

#### 验收标准

1. WHEN 功能变更任务完成 THEN 系统应询问用户是否删除功能分支
2. WHEN 用户确认删除分支 THEN 系统应删除本地和远程的功能分支
3. WHEN 删除分支前 THEN 系统应确认 MR 已合并或用户明确放弃该分支
4. WHEN 删除分支失败 THEN 系统应记录错误并通知用户
5. WHEN 用户选择保留分支 THEN 系统应跳过删除操作

### 需求 5: 代码工具管理

**用户故事:** 作为系统管理员，我想要管理和监控代码工具的状态，以便确保系统正常运行

#### 验收标准

1. WHEN 系统启动 THEN 系统应检测配置的代码工具是否可用
2. WHEN 代码工具不可用 THEN 系统应记录错误并通知管理员
3. WHEN 查询工具状态 THEN 系统应返回当前使用的工具类型、版本和可用性
4. WHEN 代码工具执行失败 THEN 系统应记录详细错误信息并尝试降级处理
5. WHEN 更新工具配置 THEN 系统应验证新配置的有效性后再应用

### 需求 6: 前端交互优化

**用户故事:** 作为用户，我想要更流畅直观的前端界面，以便更高效地完成任务

#### 验收标准

1. WHEN 用户提交任务 THEN 系统应实时显示任务执行进度和状态
2. WHEN 任务执行中 THEN 系统应通过 WebSocket 实时推送日志和代码变更
3. WHEN 显示代码变更 THEN 系统应使用 diff 视图高亮显示修改内容
4. WHEN 任务完成 THEN 系统应清晰展示执行结果和下一步操作建议
5. WHEN 任务失败 THEN 系统应显示详细错误信息和可能的解决方案

## 非功能性需求

### NFR-1: 性能
- 代码工具执行响应时间: < 30 秒（取决于具体工具）
- 工具切换延迟: < 1 秒
- 项目初始化时间: < 10 秒（不含 Git 克隆时间）

### NFR-2: 可靠性
- 代码工具调用失败时自动重试（最多 3 次）
- 工具配置验证机制，防止无效配置
- 工具执行超时保护（默认 60 秒）

### NFR-3: 安全性
- 代码工具 API Key 加密存储
- 敏感配置信息不记录到日志
- 项目删除操作需要二次确认

### NFR-4: 可用性
- 工具配置界面简洁直观
- 提供工具状态监控面板
- 错误提示清晰，提供解决建议

### NFR-5: 可扩展性
- 支持切换不同的代码工具（qodercli、Cursor、Copilot 等）
- 代码工具采用适配器模式，便于添加新工具支持
- 支持添加更多项目模板和脚手架
- 配置文件支持扩展字段，便于未来功能增强

## 技术约束

- 虚拟机: 需要配置 Git 访问权限，能够克隆私有仓库
- 代码工具: 
  - qodercli（默认）: 需要在虚拟机上安装并登录 Qoder 账号
  - 其他工具: 根据具体工具要求配置（如 API Key、环境变量等）
  - 工具配置: 通过环境变量或配置文件指定工具类型和参数
- 配置文件: 使用 JSON 或 YAML 格式存储工具配置
- 适配器接口: 所有代码工具必须实现统一的接口规范
