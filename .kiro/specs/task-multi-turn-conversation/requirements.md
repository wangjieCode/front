# 任务多轮对话 - 需求文档

## 简介

本功能允许用户与 AI 代码助手进行多轮对话来完成一个任务,而不是一次性提交完整的任务描述。用户可以逐步澄清需求、提供反馈、调整方向,AI 也可以主动询问问题以更好地理解任务目标。这种交互式的工作方式更符合真实的协作场景,提高任务完成的准确性和用户满意度。

## 术语表

- **对话会话 (Conversation Session)**: 围绕一个任务目标的完整多轮对话过程
- **对话轮次 (Turn)**: 一次用户输入和 AI 响应的完整交互
- **上下文 (Context)**: 对话历史、任务状态、代码变更等信息的集合
- **暂停点 (Pause Point)**: AI 主动暂停执行,等待用户确认或提供更多信息的时刻
- **任务状态 (Task State)**: 任务在对话过程中的状态,包括 planning(规划中)、executing(执行中)、paused(已暂停)、completed(已完成)、failed(失败)

## 需求

### 需求 1: 对话会话管理

**用户故事**: 作为用户,我希望能够创建和管理多轮对话会话,以便与 AI 协作完成复杂任务

#### 验收标准

1. WHEN 用户创建新任务 THEN 系统应创建一个新的对话会话并分配唯一的会话 ID
2. WHEN 对话会话创建 THEN 系统应初始化会话上下文,包括项目信息、初始任务描述和空的对话历史
3. WHEN 用户查看任务列表 THEN 系统应显示所有对话会话及其当前状态
4. WHEN 用户选择一个会话 THEN 系统应加载该会话的完整对话历史和上下文
5. WHEN 对话会话结束 THEN 系统应保存完整的对话记录和最终任务结果

### 需求 2: 多轮消息交互

**用户故事**: 作为用户,我希望能够在任务执行过程中随时与 AI 对话,以便澄清需求或调整方向

#### 验收标准

1. WHEN 用户发送消息 THEN 系统应将消息添加到对话历史并触发 AI 响应
2. WHEN AI 生成响应 THEN 系统应将响应添加到对话历史并通过 WebSocket 实时推送给前端
3. WHEN 对话历史更新 THEN 系统应保持消息的时间顺序和完整性
4. WHEN 用户在任务执行中发送消息 THEN 系统应暂停当前执行,处理用户消息后再继续
5. WHEN 消息包含代码或文件引用 THEN 系统应正确解析并在上下文中关联相关资源

### 需求 3: AI 主动询问

**用户故事**: 作为用户,我希望 AI 在需要更多信息时能够主动询问,而不是基于不完整的信息做出错误决策

#### 验收标准

1. WHEN AI 遇到模糊的任务描述 THEN 系统应生成澄清问题并暂停执行等待用户回答
2. WHEN AI 需要在多个方案中选择 THEN 系统应列出选项并请求用户决策
3. WHEN AI 发现潜在风险 THEN 系统应警告用户并询问是否继续
4. WHEN 用户回答 AI 的问题 THEN 系统应将答案整合到上下文中并继续执行
5. WHEN AI 询问超时未收到回复 THEN 系统应使用默认选项或标记任务为需要用户介入

### 需求 4: 上下文持久化

**用户故事**: 作为用户,我希望对话上下文能够被保存,以便我可以随时中断和恢复对话

#### 验收标准

1. WHEN 对话进行中 THEN 系统应实时保存对话历史到数据库
2. WHEN 用户关闭浏览器 THEN 系统应保留对话会话状态
3. WHEN 用户重新打开任务 THEN 系统应恢复完整的对话历史和上下文
4. WHEN 对话上下文包含代码变更 THEN 系统应保存变更的快照和版本信息
5. WHEN 系统重启 THEN 系统应能够从持久化存储中恢复所有活跃的对话会话

### 需求 5: 任务状态流转

**用户故事**: 作为用户,我希望清楚地了解任务在对话过程中的状态变化

#### 验收标准

1. WHEN 任务创建 THEN 系统应将任务状态设置为 planning(规划中)
2. WHEN AI 开始执行代码变更 THEN 系统应将任务状态更新为 executing(执行中)
3. WHEN AI 等待用户输入 THEN 系统应将任务状态更新为 paused(已暂停)
4. WHEN 任务完成所有目标 THEN 系统应将任务状态更新为 completed(已完成)
5. WHEN 任务执行失败 THEN 系统应将任务状态更新为 failed(失败)并记录失败原因

### 需求 6: 对话历史展示

**用户故事**: 作为用户,我希望看到清晰的对话历史,包括谁说了什么以及何时说的

#### 验收标准

1. WHEN 显示对话历史 THEN 系统应按时间顺序展示所有消息
2. WHEN 显示用户消息 THEN 系统应使用不同的样式区分用户和 AI 消息
3. WHEN 显示 AI 消息 THEN 系统应包含思考过程、工具调用和执行结果
4. WHEN 对话历史很长 THEN 系统应支持滚动查看并自动滚动到最新消息
5. WHEN 消息包含代码块 THEN 系统应使用语法高亮显示

### 需求 7: 输入框和发送控制

**用户故事**: 作为用户,我希望有一个方便的输入界面来发送消息给 AI

#### 验收标准

1. WHEN 任务处于可交互状态 THEN 系统应显示消息输入框
2. WHEN 用户输入消息 THEN 系统应支持多行文本和 Markdown 格式
3. WHEN 用户按下发送按钮或 Ctrl+Enter THEN 系统应发送消息给 AI
4. WHEN AI 正在处理消息 THEN 系统应禁用输入框并显示处理中状态
5. WHEN 任务已完成或失败 THEN 系统应禁用输入框或提示用户创建新任务

### 需求 8: 代码变更关联

**用户故事**: 作为用户,我希望在对话中能够看到 AI 做出的代码变更,并能够讨论这些变更

#### 验收标准

1. WHEN AI 执行代码变更 THEN 系统应在对话流中插入代码变更卡片
2. WHEN 用户点击代码变更卡片 THEN 系统应展开显示详细的 diff 视图
3. WHEN 用户对代码变更有疑问 THEN 用户应能够在对话中引用特定的变更
4. WHEN AI 解释代码变更 THEN 系统应在消息中高亮显示相关的代码片段
5. WHEN 代码变更被回滚 THEN 系统应在对话历史中标记该变更已失效

### 需求 9: 对话分支和回溯

**用户故事**: 作为用户,我希望能够回到对话的某个历史点,尝试不同的方向

#### 验收标准

1. WHEN 用户选择历史消息 THEN 系统应提供"从此处继续"的选项
2. WHEN 用户从历史点继续 THEN 系统应创建新的对话分支
3. WHEN 存在多个对话分支 THEN 系统应允许用户在分支间切换
4. WHEN 切换对话分支 THEN 系统应恢复该分支的上下文和代码状态
5. WHEN 用户合并分支 THEN 系统应提示可能的冲突并协助解决

### 需求 10: 实时协作提示

**用户故事**: 作为用户,我希望看到 AI 当前正在做什么,即使它还没有完成

#### 验收标准

1. WHEN AI 正在思考 THEN 系统应显示"正在思考..."的动画提示
2. WHEN AI 正在调用工具 THEN 系统应显示工具名称和执行进度
3. WHEN AI 正在生成代码 THEN 系统应显示"正在编写代码..."的提示
4. WHEN AI 等待外部服务响应 THEN 系统应显示等待状态和预计时间
5. WHEN 操作超时 THEN 系统应提示用户并提供重试或取消选项

## 非功能性需求

### NFR-1: 性能
- 消息发送响应时间: < 200ms
- 对话历史加载时间: < 1 秒(100 条消息内)
- 上下文切换延迟: < 500ms
- WebSocket 消息推送延迟: < 100ms

### NFR-2: 可靠性
- 对话上下文自动保存间隔: 每 5 秒或每条新消息
- WebSocket 断线自动重连(最多 5 次)
- 消息发送失败自动重试(最多 3 次)
- 数据库事务保证对话历史完整性

### NFR-3: 可用性
- 支持 Markdown 格式的消息输入
- 支持代码块语法高亮
- 支持消息搜索和过滤
- 支持导出对话历史为文本或 JSON

### NFR-4: 可扩展性
- 支持插件式的消息类型扩展
- 支持自定义 AI 提示词模板
- 支持多种 AI 模型切换
- 支持对话历史的归档和清理策略

## 技术约束

- 后端: Node.js + TypeScript + Express
- 数据库: 使用现有的数据存储方案(文件系统或数据库)
- 实时通信: WebSocket (已有实现)
- AI 服务: 集成现有的 NeovateAIService
- 前端: React + TypeScript + Vite
